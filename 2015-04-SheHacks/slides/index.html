0. Install Meteor

Set up Meteor, Node and Mongo distributions. 

$ curl https://install.meteor.com/ | sh

To follow along, also clone the shehacksnyc repo 

$ git clone https://github.com/MeteorHudsonValley/shehacksnyc.git

Meteor is relatively stable on *nix systems. A preview version was just released for Windows *.
Once you clone the repo, you can grab the version for a given step by simply checking out the tag. The tag for each step will be clearly marked in the slide title. So a title with 'tags/step-1' implies you can switch to code for that step using 'git checkout tags/step-1' on your local cloned repo.

1. Scaffold an app tags/step-1

Meteor creates the basic boilerplate code for you. 

$ meteor create webapp

Open an editor. Let's explore the code. 

ls -al ./webapp
webapp.js     // app processing (contains data helpers, event handlers)
webapp.html   // app view (contains body, header, templates)
webapp.css    // app styles
.meteor       // meteor configuration, local db

Directory hierarchy, file names have meaning. The .meteor directory is managed by Meteor and should not be edited manually.

1a. Run The App

Meteor's isobuild toolchain creates a runnable app. 

$ cd webapp
$ meteor

=> Started proxy.                             
=> Started MongoDB.                           
=> Started your app.                          

=> App running at: http://localhost:3000/

Open a browser. Let's explore the app.

Boilerplate app shows a basic view (template), event handling (click) features. No styling (yet).
Use 'meteor run' explicitly to run against a specific platform e.g., 'meteor run android' runs app in an Android emulator.
Use 'meteor reset' to clear a local database and 'meteor mongo' to open a MongoDB console to it.

1b. Deploy The App

Meteor's isobuild toolchain also uploads the runnable app. 

$ meteor deploy shehacksnyc.meteor.com

2. Templates tags/step-2

are 'blueprints' (with data placeholders) for modular UI


the templating system has three parts:

Spacebars (language)
to add logic and data to views 		Blaze (engine)
to activate and renders templates into DOM		 Tracker (library)
to trigger reactive updates on data changes

Templates are passive HTML structures with placeholders for data.
Template helpers bind data 'contexts' dynamically [ref], before it is rendered into DOM.
Template tags (#with, #each) can be used to set the data context explicitly.
Template events declare handlers for DOM events associated with that template.

3. Collections tags/step-3

persist application data to a (MongoDB) database

MongoDB (server) full-featured db, utilities MiniMongo (client) in-memory db, consistent API DDP (protocol) transfers data, syncs both db reactively


Collections expose a consistent API on both client and server sides.
Template helpers often query Collections, returning reactive data cursors
Collections are insecure and autopublish by default

3a. About MongoDB

MongoDB is a document database, useful for storing unstructured data (flexible schema)

Open a terminal. Let's explore the local Mongo db. 

$ meteor mongo
MongoDB shell version: 2.6.7
connecting to: 127.0.0.1:3001/meteor
meteor:PRIMARY>

Data is persisted in a JSON-like format (name-value pairs)
Data is grouped into "Collections" (analogous to tables in RDBMS)
Use replica sets to support fault tolerance (vertical scaling)
Use sharding to support load balancing (horizontal scaling)

3b. MongoDB Primer (try a tutorial)

First, some basic commands:

db show currently active db
show databases show all (non-empty) dbs on this server
show collections show all collections in current db
use mydb make 'mydb' the current (active) db

db.help() list all methods that can be called on db
db.createCollection('myColl') create the 'myColl' collection on current db
db.showCollections() show all collections in current db
db.dropDatabase() delete the db
db.myColl.drop() delete th 'myColl' collection
db.myColl.dropIndexes() delete all indexes on the 'myColl' collection
db.myColl.getDB() get the db object associated with the 'myColl' collection

3c. MongoDB Primer (try a tutorial)

Next, some collection operations (on 'myColl')

db.myColl.count() number of documents in collection
db.myColl.insert(doc) add document to collection
db.findOne(query) return first document matching query
db.myColl.find(query) return all documents matching query
db.myColl.update(query, object[.., upsert_flag, multi_flag]) updatematching documents (insert if needed) (first match vs. all)
db.myColl.remove(query) remove all matching documents

8. Temporary UI State tags/step-8

Use 'Session' variables to share transient UI state


Session is a convenient place to store temporary UI state, and can be used in helpers just like a collection



This is a good time to talk about Reactivity: Reactive Computations and Reactive Data Sources.
A Reactive Data Source knows when its contents change, and can notify listeners of that change. Ex: Session, Collection queries, Meteor.user
A Reactive Computation is one that automatically recomputes itself when a contained Reactive Data Source changes. Ex: Template methods, Tracker.autorun, Blaze.render..
Here Session is used to make 'hideCompleted' state reactive for template use.

10. Security w/ Methods tags/step-10


By default Meteor is in 'insecure' mode, allowing clients to directly modify the database. Let's fix this. 

meteor remove insecure

Meteor.methods enables RPC-like behavior with latency compensation for fast but secure updates.

Latency Compensation with Meteor.call(). Methods is executed securely on server. Meanwhile, method can 'predict' outcome on local client to speed up response. Conflicts are resolved when server computation completes and data is synced.

11. Publish / Subscribe tags/step-11


By default Meteor is in 'autopublish' mode, synching all data between client and server. Let's fix this. 

meteor remove autopublish

Publish (server) and Subscribe (client) allow data transfers to be regulated for efficiency and context.
